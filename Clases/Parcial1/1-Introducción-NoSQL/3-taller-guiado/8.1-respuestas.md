# Selección de Bases de Datos: SQL vs NoSQL

## Ejercicio 1: Gestión Universitaria
**Escenario:** Gestión de matrículas, asignaturas, profesores y actas de notas. Se requiere consistencia fuerte y transacciones seguras.

### Solución: Base de Datos Relacional (SQL)

**Justificación:**
* **Integridad ACID:** En un entorno académico, los datos son críticos. Las bases relacionales garantizan la atomicidad: si se guarda un acta, se guardan todas las notas o ninguna.
* **Consistencia Fuerte:** No es aceptable la "consistencia eventual". Si un alumno se matricula, el cupo debe descontarse inmediatamente para evitar conflictos.
* **Relaciones Estrictas:** Existen reglas de negocio inamovibles (Integridad Referencial) que SQL valida nativamente.

> **Tecnologías:** PostgreSQL, Oracle, MySQL.

## Ejercicio 2: Aplicación Móvil (Sesiones)
**Escenario:** Almacenamiento de preferencias, tokens de sesión y configuraciones temporales. Acceso inmediato y borrado automático.

### Solución: Base de Datos Clave–Valor (NoSQL)

**Justificación:**
* **Velocidad de Acceso (O(1)):** Al buscar por "token", el acceso es directo e instantáneo (hash map). No se requieren *joins* ni recorridos complejos.
* **TTL (Time-To-Live):** Característica nativa para datos efímeros. Los datos expiran y se auto-eliminan tras el tiempo de inactividad sin procesos de limpieza externos.
* **Simplicidad:** El modelo de datos es plano y sin relaciones.

> **Tecnologías:** Redis, Amazon DynamoDB.

---

## Ejercicio 3: Sensores Industriales (IoT)
**Escenario:** Millones de registros diarios, volumen en aumento y consultas de resúmenes analíticos por grupos (día, planta, tipo).

### Solución: Base de Datos Columnar (NoSQL)

**Justificación:**
* **Escritura Masiva:** Diseñadas para soportar una ingesta masiva de datos (millones de eventos/segundo) que saturaría a una base SQL.
* **Consultas Analíticas (OLAP):** Almacenan datos por columnas. Calcular el promedio de temperatura de un año es muy rápido porque solo se lee la columna de "temperatura", ignorando el resto del disco.
* **Escalabilidad Horizontal:** Permite añadir nodos fácilmente al clúster conforme crecen los datos.

> **Tecnologías:** Apache Cassandra, ScyllaDB, Google BigTable.

## Ejercicio 4: Gestión de Contenidos (CMS)
**Escenario:** Artículos con campos variables (video, galerías, texto) y cambios frecuentes en el modelo de datos.

### Solución: Base de Datos Documental (NoSQL)

**Justificación:**
* **Schema-less (Sin esquema fijo):** Permite polimorfismo. Un documento puede tener video y otro solo texto sin romper la base de datos ni requerir migraciones (`ALTER TABLE`).
* **Datos Agregados:** La información relacionada (etiquetas, metadatos básicos) se guarda en el mismo documento JSON, evitando uniones costosas.
* **Iteración Rápida:** Ideal para aplicaciones donde los requisitos del producto cambian semanalmente.

> **Tecnologías:** MongoDB, Couchbase, Firestore.

## Ejercicio 5: Sistema Bancario
**Escenario:** Registro de transferencias entre cuentas. Prioridad absoluta en que no se pierda dinero y consistencia total ante fallos.

### Solución: Base de Datos Relacional (SQL)

**Justificación:**
* **Seguridad Transaccional:** Previene problemas de concurrencia (como el "doble gasto") mediante bloqueos de fila y aislamiento de transacciones.
* **Atomicidad y Durabilidad:** Si una transferencia falla a la mitad, el sistema hace *rollback* automático. El dinero nunca desaparece ni se duplica.
* **Modelo Estable:** La relación Cliente-Cuenta-Movimiento es estructuralmente rígida y no requiere la flexibilidad de NoSQL.

> **Tecnologías:** PostgreSQL, IBM Db2, Microsoft SQL Server.

---

## Ejercicio 6: Red Social
**Escenario:** Detección de comunidades, relaciones indirectas y conexiones sospechosas entre entidades.

### Solución: Base de Datos de Grafos (NoSQL)

**Justificación:**
* **Relaciones como Entidades:** La conexión entre nodos es tan importante como el dato mismo.
* **Eficiencia en Recorridos:** SQL sufre con *joins* recursivos ("amigo del amigo del amigo"). Las bases de grafos resuelven esto instantáneamente recorriendo punteros físicos.
* **Detección de Patrones:** Ideal para descubrir fraudes, bucles o clusters de usuarios interconectados.

> **Tecnologías:** Neo4j, Amazon Neptune.

---

## Ejercicio 7: Buscador Semántico
**Escenario:** Encontrar documentos similares a una descripción, aunque no coincidan las palabras exactas.

### Solución: Base de Datos Vectorial (NoSQL)

**Justificación:**
* **Búsqueda Semántica:** A diferencia de SQL (`LIKE %texto%`), aquí se busca por significado/contexto utilizando vectores matemáticos.
* **Embeddings:** Permite comparar qué tan "cerca" están dos conceptos (ej: "coche" y "automóvil") en un espacio multidimensional.
* **Uso de IA:** Es la base de datos estándar para integraciones con LLMs y sistemas de recomendación modernos.

> **Tecnologías:** Pinecone, Weaviate, Milvus, pgvector.

---

## Ejercicio 8: Sistema Corporativo Simple
**Escenario:** Empleados, departamentos y jerarquías simples. Volumen moderado y relaciones estables.

### Solución: Base de Datos Relacional (SQL)

**Justificación:**
* **Simplicidad y Estándar:** Para problemas estructurados con volumen controlado, SQL es la solución más madura, fácil de mantener y con mayor disponibilidad de talento.
* **Integridad de Datos:** Asegura relaciones válidas (un empleado no puede estar en un departamento inexistente).
* **Consultas Predecibles:** No se requiere la complejidad ni las ventajas de escalado masivo de NoSQL.

> **Tecnologías:** MySQL, MariaDB, PostgreSQL.
